# Тема: Система контролю версій Git та співпраця через GitHub

Git — це розподілена система контролю версій (VCS, Version Control System), створена Лінусом Торвальдсом у 2005 році для розробки ядра Linux. Сьогодні Git є де-факто стандартом для контролю версій у програмному забезпеченні та використовується мільйонами розробників по всьому світу.
Система контролю версій дозволяє відстежувати зміни в коді з часом, повертатися до попередніх версій, працювати над проектом командою без конфліктів, експериментувати з новими функціями в ізольованих гілках, та багато іншого. Git зберігає повну історію проекту, що дозволяє відповісти на питання "хто, що, коли і чому змінив" у будь-який момент часу.

### Контрольні питання

1. Основи Git

Яка різниця між Git та GitHub?
 • Git — система контролю версій, локальна.
 • GitHub — сервіс для хостингу Git-репозиторіїв онлайн.

Що означає “розподілена” система контролю версій?
 • Кожен користувач має повну копію репозиторію, можна працювати автономно.

Яка різниця між git init та git clone?
 • git init — створює новий локальний репозиторій.
 • git clone — копіює існуючий репозиторій з віддаленого сервера.

2. Staging Area та Commits

Що таке Staging Area і навіщо вона потрібна?
 • Проміжна зона для підготовки змін перед комітом.

Яка різниця між git add . та git add -A?
 • git add . — додає зміни в поточній директорії.
 • git add -A — додає всі зміни у репозиторії, включно з видаленнями.

Як написати гарне commit message?
 • Короткий опис (50 символів) + детальний опис, якщо потрібно; дієслово у теперішньому часі.

Як виправити останній коміт?

git commit --amend

3. Гілки та злиття

Що таке гілка в Git і чому вони “легковажні”?
 • Гілка — незалежна лінія розвитку; створювати та видаляти її легко.

Яка різниця між git merge та git rebase?
 • merge — створює коміт злиття, зберігаючи історію.
 • rebase — переносить коміти на нову базу, роблячи історію прямою.

Що таке fast-forward merge?
 • Злиття без коміту merge, коли гілка можна просто «проштовхнути» вперед.

Коли виникають merge конфлікти і як їх уникнути?
 • Конфлікт при редагуванні одних рядків у різних гілках.
 • Уникати: часті коміти, pull перед роботою, координувати зміни.

Чому не можна робити rebase публічних комітів?
 • Бо змінюється історія, що вже бачать інші, може спричинити проблеми при пуші.

4. Віддалені репозиторії

Що таке origin?
 • Псевдонім для віддаленого репозиторію, звідки клонувався проект.

Яка різниця між git fetch та git pull?
 • fetch — отримує зміни без злиття.
 • pull — отримує зміни і робить merge або rebase.

Що робить git push –force і чому це небезпечно?
 • Перезаписує історію віддаленої гілки.
 • Може стерти чужі коміти, якщо працюєш в команді.

Як видалити віддалену гілку?

git push origin --delete branch_name

5. Скасування змін

Яка різниця між git reset –soft, –mixed та –hard?
 • --soft — залишає зміни в staging.
 • --mixed — залишає зміни в робочій директорії.
 • --hard — видаляє всі зміни.

Яка різниця між git reset та git revert?
 • reset — переписує історію локально.
 • revert — створює новий коміт, який скасовує попередній.

Як скасувати git add (видалити файл зі staging)?

git restore --staged <file>

Як повернути видалений файл?

git checkout -- <file>

6. Безпека

Що не повинно потрапляти в Git репозиторій?
 • Паролі, ключі, .env файли, node_modules (залежності).

Що робити якщо випадково закомітили пароль?
 • Видалити з історії (git filter-branch або git reset --hard), змінити пароль.

HTTPS vs SSH для GitHub — що краще?
 • SSH зручніше і безпечніше для постійного доступу.
 • HTTPS зручно для разових комітів або CI.

Що таке Personal Access Token і навіщо він потрібен?
 • Альтернатива паролю для доступу до GitHub через HTTPS; потрібен для push/pull.

7. GitHub

Що таке Pull Request?
 • Запит на злиття твоєї гілки в іншу (зазвичай main), з обговоренням змін.

Яка різниця між Issues та Pull Requests?
 • Issues — для завдань, багів, ідей.
 • Pull Requests — для змін коду з обговоренням і рев’ю.

Що таке Fork і навіщо він потрібен?
 • Копія чужого репозиторію у твоєму акаунті для власних змін.

Що таке GitHub Actions?
 • CI/CD інструмент для автоматизації тестів, збірки та деплойменту.

8. Best Practices

Як часто треба робити коміти?
 • Часто, після завершення логічної зміни.

Скільки часу має жити feature branch?
 • До завершення функціоналу або задачі; після злиття можна видаляти.

Як організувати git workflow в команді?
 • Використовувати main для стабільної версії, feature branches для розробки, Pull Requests для злиття.

Чи треба комітити node_modules або інші залежності?
 • Ні, їх додають у .gitignore, бо їх легко відновити через package.json / package-lock.json.
